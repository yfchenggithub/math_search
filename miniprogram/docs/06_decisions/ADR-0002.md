# ADR-0002：双源架构——远程优先 + 本地回退

- **状态：** accepted
- **日期：** 2026-04
- **决策者：** 开发者
- **影响范围：** 搜索、建议、详情、网络层

---

## 背景

小程序面临两种极端场景：
1. **网络良好时**：用户期望最新数据（热门排序、实时更新）
2. **网络不可用时**：用户在小程序内必须有可用体验（不能白屏/卡死）

微信小程序审核要求：首次打开必须有可用内容，不能依赖网络。

---

## 决策

所有核心功能（搜索、建议、详情）采用**远程优先 + 本地回退**模式：

```
远程 API 请求
  ├── 成功 → 使用远程数据
  └── 失败 → ENABLE_LOCAL_FALLBACK?
                ├── true  → 使用本地数据
                └── false → 返回错误
```

通过 `config/api.ts` 中的开关控制：
- `USE_REMOTE_API`: 全局远程开关
- `ENABLE_LOCAL_FALLBACK`: 失败时是否回退本地

---

## 备选方案

### 方案 A：纯远程（无本地）

- 优点：数据实时，无需打包索引
- 缺点：无网络时不可用，审核不通过
- 否决原因：微信审核要求 + 用户体验

### 方案 B：纯本地（无远程）

- 优点：永远可用，无网络依赖
- 缺点：数据更新需发版，无法个性化排序/推荐
- 否决原因：长期需要远程能力（热门、用户行为）

### 方案 C：本地优先 + 远程更新（最终一致性）

- 优点：首屏最快
- 缺点：用户可能先看到旧数据再刷新，体验闪变
- 否决原因：搜索结果闪变比等待更差

---

## 影响

- 所有数据获取入口必须经过 facade 层（search-engine.ts 中的 `suggestWithFacade` / `searchWithFacade`）
- 远程和本地输出必须对齐 ViewModel 结构
- 网络超时设置为 10s（`API_CONFIG.timeout`），超时视为失败触发回退

---

## 回滚方式

- 设置 `USE_REMOTE_API: false` → 全局回退纯本地模式
- 设置 `ENABLE_LOCAL_FALLBACK: false` → 远程失败不兜底（仅调试用）
