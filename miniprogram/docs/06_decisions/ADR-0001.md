# ADR-0001：索引格式从多层 JSON 迁移到单文件 JS Bundle

- **状态：** accepted
- **日期：** 2026-04-16
- **决策者：** 开发者
- **影响范围：** 搜索索引加载、构建脚本、数据目录结构

---

## 背景

原始设计（见 `data/readme`）定义了 4 层数据文件：

```
data/index/core_index.json     # 词 → 结论ID 映射
data/index/suggest_index.json  # 输入 → 推荐关键词
data/index/rank_index.json     # ID → 排序权重
data/content/*.json            # ID → 完整内容
```

实际开发中发现：
1. 微信小程序对 JSON 文件的支持有限，大 JSON 解析性能差
2. 4 个独立文件意味着 4 次加载，增加启动延迟
3. rank_index 的权重数据与 core_index 的文档元数据高度耦合，频繁跨文件查询
4. 构建脚本更倾向于输出单一制品

---

## 决策

将所有索引数据合并为单个 **JavaScript module**：`data/index/search_bundle.js`

结构：
```js
module.exports = {
  version: 1,
  generatedAt: "2026-...",
  fieldMaskLegend: { title: 1, alias: 2, ... },
  docs: { ... },          // ID → SearchDoc（含 rank/hotScore）
  termIndex: { ... },     // 词 → 倒排列表
  prefixIndex: { ... },   // 前缀 → 倒排列表
  suggestions: [ ... ],   // 补全建议
}
```

优势：
- 单文件 `require()`，一次加载
- JS module 解析由微信原生引擎处理，性能远超 `JSON.parse()`
- docs 合并了 rank 信息，消除跨文件查询
- 构建脚本统一输出，版本号内嵌

---

## 备选方案

### 方案 A：保持 4 层 JSON（原始设计）

- 优点：关注点分离清晰，可独立更新各层
- 缺点：4 次 I/O，解析开销，跨文件查询代码复杂
- 否决原因：微信环境不适合多文件 IO 模式

### 方案 B：单文件 JSON

- 优点：比 4 个文件好，结构简单
- 缺点：大 JSON parse 仍有性能问题
- 否决原因：JS module 的 `require()` 比 `JSON.parse(fs.readFileSync())` 快得多

---

## 影响

- 搜索引擎 `search-engine.ts` 直接 `require(search_bundle.js)` 而非加载多个 JSON
- 构建脚本需改造为输出 JS module 格式
- 数据目录结构简化：`data/index/` 下只有 search_bundle.js
- 内容层（content/*.js）不受影响，仍按模块独立加载

---

## 回滚方式

如需回滚到多层 JSON：
1. 从 search_bundle.js 中提取 termIndex / prefixIndex / suggestions / docs
2. 分别写入 core_index.json / suggest_index.json / rank_index.json
3. 修改 search-engine.ts 的加载逻辑为多文件模式
4. 回滚构建脚本
